# MiniSocial Specification — Часть 2 (дополнение с недостающими деталями)

Этот документ является дополнением к `MiniSocial-specification.md`.
Здесь перечислено поведение, которое реализовано в проекте, но недостаточно явно описано (или отсутствует) в исходном документе.
Документ намеренно не повторяет уже покрытую базовую функциональность.

---

## 1) Жизненный цикл UI создания поста (критично для точного воссоздания)

- По умолчанию форма создания поста **не встроена** в ленту.
- Для авторизованных пользователей отображается плавающая кнопка действия (`+`) как `#open-compose`.
- Контейнер модалки создания поста (`#compose-modal`) изначально имеет `hidden` и открывается только через JS.
- Модалка закрывается:
  - кнопкой закрытия (`#close-compose`)
  - кликом по затемнению (backdrop)
  - клавишей `Escape` (если не открыт пиксель-редактор)
- При открытии compose:
  - черновик галереи очищается
  - поповер эмодзи закрывается
  - фокус ставится в textarea
  - авто-ресайз textarea и счетчик символов пересчитываются сразу

Почему это важно: статическая всегда видимая форма "Post" не соответствует UX итогового проекта.

---

## 2) Ровно три источника изображений и как они сериализуются

Галерея в compose поддерживает ровно три пользовательских пути:

1. **Add file** -> локальный объект `File`
2. **Add URL** -> строка внешнего URL
3. **Draw 8x8** -> экспорт из пиксель-редактора в PNG-объект `File`

При отправке JS преобразует состояние галереи в динамические поля формы:

- `gallery_count`
- для каждого слота `i`:
  - `gallery_slot_{i}_kind` = `file` или `url`
  - если слот URL -> `gallery_url_{i}`
  - если слот file -> синтетический скрытый `input[type=file]` с именем `gallery_file_{i}`, заполняемый через `DataTransfer`

Важно: сервер опирается именно на эту структуру, это не только UI-деталь.

---

## 3) Правила размера файлов и валидации (точно и без двусмысленностей)

### К чему применяется лимит 300 KB

- `MAX_POST_IMAGE_BYTES` (300 KB) применяется **только к загружаемым бинарным данным**:
  - обычные загруженные файлы
  - результат рисования пиксель-арта (потому что он отправляется как PNG-файл)

### К чему лимит 300 KB **не** применяется

- URL-элементы валидируются только по:
  - схеме/хосту (`http://` или `https://` + netloc)
  - длине строки URL (`MAX_IMAGE_URL_LENGTH`, 1024)
- Сервер не загружает удаленный контент по URL для проверки размера в байтах.

### Дополнительные проверки на стороне compose

- JS предварительно валидирует URL регуляркой `^https?://\S+$`
- JS предварительно валидирует размер файла до добавления в галерею
- Сервер выполняет повторную независимую проверку (источник истины)

---

## 4) Лимиты галереи и поведение слотов

- UI разрешает максимум `MAX_IMAGES_PER_POST` (4) элемента галереи суммарно между всеми тремя источниками.
- Когда лимит достигнут, кнопки `Add file`, `Add URL` и `Draw 8x8` отключаются.
- Элементы галереи можно удалить до отправки.
- Сервер ограничивает некорректный `gallery_count` в диапазон `[0, MAX_IMAGES_PER_POST]`.
- Невалидный или отсутствующий payload слота приводит к немедленному отклонению с flash и редиректом.

---

## 5) Детали текстового поля compose, отсутствующие в основном документе

- Textarea использует серверный `maxlength`, счет в символах идет на клиенте в реальном времени.
- Визуальные состояния счетчика:
  - близко к лимиту (`max - 20`)
  - достигнут лимит
- Textarea автоматически растет до половины высоты вьюпорта (предел ~352 px), затем включает прокрутку.
- Пустой текст и превышение лимита блокируются в клиентском JS до submit; сервер повторяет валидацию.

---

## 6) Поведение кнопки лайка в рантайме (AJAX + fallback)

- Каждое действие лайка остается обычной HTML-формой для fallback без JS.
- JS перехватывает submit и вызывает `POST /api/posts/<id>/like`.
- При успехе:
  - счетчик лайков обновляется на месте
  - подпись переключается `Like` <-> `Liked`
- При ошибке fetch/исключении JS откатывается к обычной отправке формы.
- Кнопка лайка временно блокируется на время запроса.

Такой fallback важен, чтобы соответствовать реальной устойчивости реализации.

---

## 7) Детали lightbox сверх базового упоминания

- Клик по любой миниатюре галереи поста открывает полноэкранный lightbox.
- Lightbox строит набор слайдов из всех изображений галереи конкретного поста.
- Поддерживается:
  - закрытие по затемнению или кнопке закрытия
  - клавиша `Escape` для закрытия
  - клавиши `ArrowLeft` / `ArrowRight` для навигации
  - экранные кнопки предыдущего/следующего
  - счетчик слайдов показывается только при наличии нескольких изображений
- Пока lightbox открыт, прокрутка страницы блокируется (`document.body.style.overflow = "hidden"`).
- Для маленьких pixel-art изображений из blob добавляется специальный класс для pixelated upscale в lightbox.

## 8) Интеграция пиксель-редактора (avatar и compose — разные потоки)

- Один общий модальный редактор поддерживает две цели сохранения:
  - `avatar`
  - `compose`
- Черновики compose и avatar не полностью общие:
  - у compose есть отдельный буфер в сессии (`composeBuffer`)
  - для avatar перед редактированием текущая аватарка загружается в сетку 8x8
- Поведение сохранения зависит от цели:
  - `avatar` -> экспортированный PNG помещается в скрытую форму аватара и отправляется сразу
  - `compose` -> экспортированный PNG добавляется в галерею как file-элемент; прямой POST сразу не выполняется
- Редактор использует pointer events (`pointerdown/move/up`) и явную поклеточную отрисовку.

---

## 9) Детали поведения emoji picker для точного соответствия

- Emoji picker загружается как web component из CDN ESM.
- Popover переключается отдельной кнопкой с синхронизацией `aria-expanded`.
- Вставка эмодзи учитывает оставшийся лимит `maxlength` textarea и вставляет в текущую позицию курсора.
- Тема picker синхронизируется с dark mode через `MutationObserver`, отслеживающий изменение класса `html`.
- Клик вне обертки emoji в compose закрывает popover.

---

## 10) Контроль сессии на защищенных маршрутах

Помимо проверок во время логина, каждый защищенный маршрут повторно проверяет пользователя сессии в БД:

- если пользователь из сессии отсутствует в БД:
  - сессия очищается
  - пользователь перенаправляется на login с сообщением "session no longer valid"
- если пользователь есть, но архивирован:
  - сессия очищается
  - пользователь перенаправляется на login с сообщением об архивном аккаунте
- `username`/`role` в сессии обновляются из БД на каждом защищенном запросе

Это более строгое поведение, чем разовая проверка только при входе, и его нужно явно фиксировать в спецификации.

---

## 11) Тонкие правила рендеринга автора и поста

- И архивированные, и удаленные авторы в строке автора отображаются как `user deleted`.
- Если ссылка на автора отсутствует и состояние не сопоставлено, используется запасная подпись `unknown user`.
- Галерея поста в ленте рендерится только если:
  - `author_state == "active"`
  - у поста есть элементы галереи
- Маршрут blob-изображений поста (`/post-image/...`) отдельно проверяет `author_state == "active"` перед отдачей байтов.

---

## 12) UX-детали переключателя регистрации в админке

- Контрол регистрации реализован как switch-кнопка, а не обычный checkbox.
- Клик по switch заполняет скрытый `action` (`on`/`off`) и автоматически отправляет форму.
- Контрол видим только при `can_manage_master_controls` (роль master).

---

## 13) Уточнение: реализованное поведение отличается от типичных предположений

Эти пункты важны при воссоздании по документам, потому что их легко неверно предположить:

- Нет активного потока с "всегда видимой формой поста"; compose работает через модалку + FAB.
- `Draw 8x8` не является отдельным типом медиа на бэкенде; это обычный загружаемый PNG-файл.
- URL-изображения хранятся как URL и рендерятся браузером напрямую; бэкенд их не проксирует и не скачивает.
- Лимит в байтах относится к загружаемым blob, а не к целям URL.

---

## 14) Критичный чеклист для воссоздания (только Часть 2)

Используйте этот короткий чеклист вместе с Частью 1:

- Воссоздать compose-модалку, скрытую по умолчанию, и открытие через FAB.
- Реализовать все три источника галереи и динамическую сериализацию полей формы.
- Применять лимиты в байтах только к file payload (включая файлы из pixel export).
- Для URL оставить только валидацию схемы и длины.
- Повторить AJAX-лайк с fallback на submit без JS.
- Реализовать вставку эмодзи в каретку с учетом `maxlength`.
- Реализовать управление lightbox с клавиатуры/мыши и блокировку прокрутки.
- Сохранить раздельные пути сохранения pixel editor для avatar и compose.
- Повторно валидировать активность пользователя сессии на каждом защищенном маршруте.
