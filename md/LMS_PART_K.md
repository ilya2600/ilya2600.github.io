# Часть K: Ограничения удаления, архивации и роли менеджера (v2)

В Части J вы добавили страницы «Мой прогресс» и «Статистика» и блоки недавних занятий и заданий на дашборде.

**Что уже есть после Частей B и D.** В Части B вы сделали: вход только для пользователей с `archived_at IS NULL`, функцию `current_user()` с учётом архива. В Части D вы добавили: страницу «Пользователи», кнопки «Архивировать» и «Восстановить», маршруты архивации и восстановления, а также маршрут полного удаления пользователя с кнопкой «Удалить навсегда» для любого пользователя с ролью не admin. Таблица `users` уже содержит колонку `archived_at` (она была в Части A).

**Что меняем и добавляем в Части K.** В этой части вы не переписываете архив и вход с нуля, а только:

1. **Проверяете** (при необходимости поправляете), что вход и списки «добавить в класс» по‑прежнему исключают архивированных.
2. **Добавляете в маршрут архивации** проверку: нельзя архивировать пользователя с ролью `admin`.
3. **Ограничиваете удаление:** удалять можно только уже заархивированного пользователя; в представлении удаления добавляете проверку `archived_at`, в шаблоне кнопку «Удалить навсегда» показываете только для архивированных.
4. **Ограничиваете роль менеджера при создании пользователя:** менеджер не может назначать роль «менеджер» (только мастер может); в форме создания для менеджера скрываете опции «Менеджер» и «Мастер», на бэкенде при попытке создать менеджера подменяете роль или отклоняете.

Итого: K1 — проверка, K2 — одна проверка в архивации, K3 — изменение логики и кнопки удаления, K4 — ограничение роли менеджера.

## Видимо / проверяемо

После выполнения Части K:

- Заархивированный пользователь не может войти; в списке «Ученики не в классе» на странице класса он не отображается. После восстановления снова может войти и появляется в списках.

- Кнопка «Удалить навсегда» видна **только** у архивированных пользователей. Попытка удалить активного пользователя (например, обойдя форму) завершается сообщением «Сначала архивируйте».
- Менеджер при создании пользователя видит в выборе роли только «Ученик» и «Учитель»; мастер видит все четыре роли, в том числе «Менеджер».

## K1. Проверка: вход и списки без архивированных

После Части B вход уже использует `archived_at IS NULL`; после Части G списки для «добавить ученика в класс» тоже должны учитывать только неуархивированных. Здесь нужно только **проверить** и при необходимости поправить.

### Шаг K1.1. Колонка `archived_at`

В `db.py` в `init_db()` в определении таблицы `users` должно быть поле `archived_at TEXT`. Оно было добавлено ещё в Части A. Если по какой‑то причине его нет (например, вы начали проект с другой схемы), добавьте в `CREATE TABLE users (...)` строку `archived_at TEXT,` и при необходимости выполните миграцию (`ALTER TABLE users ADD COLUMN archived_at TEXT;`).

### Шаг K1.2. Вход только для неуархивированных

В `views_auth.py` в `login_view()` при выборке пользователя по логину должен использоваться фильтр по архиву. Проверьте, что запрос выглядит так:

```python
user = conn.execute(
    "SELECT * FROM users WHERE username = ? AND archived_at IS NULL LIMIT 1",
    (username,),
).fetchone()
```

Если у вас было без `AND archived_at IS NULL`, добавьте это условие.

### Шаг K1.3. Списки «добавить в класс» без архивированных

В `views_classes.py` при формировании списка «все ученики, не входящие в данный класс» и при проверке ученика при добавлении в класс должны участвовать только пользователи с `archived_at IS NULL`. Проверьте запросы. Пример для списка «ученики не в классе»:

```sql
SELECT id, username
FROM users
WHERE role = 'student' AND archived_at IS NULL
AND id NOT IN (SELECT student_id FROM class_students WHERE class_id = ?)
ORDER BY username
```

При добавлении ученика в класс проверка пользователя по `id` и `role = 'student'` должна включать `AND archived_at IS NULL`. Если этого нет — добавьте.

## K2. Архивация: запрет архивировать мастера

В Части D маршрут архивации уже устанавливает `archived_at` для выбранного пользователя. Нужно только **добавить проверку**: если у пользователя роль `admin`, архивировать его нельзя.

### Шаг K2.1. Проверка в `admin_user_archive_view`

В `views_admin.py` в функции архивации (например, `admin_user_archive_view(user_id)`):

1. После выборки пользователя по `user_id` загружайте не только `id`, но и `role`: например, `SELECT id, role FROM users WHERE id = ?`.
2. Если пользователь не найден — `abort(404)`.
3. **Если `u["role"] == "admin"`** — не выполнять обновление, закрыть соединение, вывести `flash("Нельзя заархивировать мастер‑аккаунт.")` и сделать `redirect(url_for("admin_users"))`.
4. Иначе — как раньше: установить `archived_at = текущая дата/время` и редирект.

Пример фрагмента (после получения `u`):

```python
if u["role"] == "admin":
    conn.close()
    flash("Нельзя заархивировать мастер‑аккаунт.")
    return redirect(url_for("admin_users"))

now = datetime.utcnow().strftime("%Y-%m-%dT%H:%M:%SZ")
conn.execute("UPDATE users SET archived_at = ? WHERE id = ?", (now, user_id))
# ... commit, close, flash, redirect
```

### Шаг K2.2. Кнопки в шаблоне

В Части D в `admin_users.html` уже должны быть: для архивированного — «Восстановить», для активного не-admin — «Архивировать»; для admin кнопки архивации/восстановления не показываются. Проверьте, что так и есть. Менять шаблон для K2 не обязательно, если логика уже совпадает с этой.

## K3. Удаление только архивированных пользователей

В Части D маршрут удаления позволял удалить любого пользователя с ролью не `admin`. По требованиям ЛМС удалять можно **только заархивированного** пользователя: сначала архив, потом при необходимости «Удалить навсегда». В этом блоке вы меняете и представление, и шаблон.

### Шаг K3.1. Проверка в `admin_user_delete_view`

В `views_admin.py` в функции удаления (например, `admin_user_delete_view(user_id)`):

1. Загрузите пользователя по `user_id`, включая поля `role` и `archived_at`:  
   `SELECT id, username, role, archived_at FROM users WHERE id = ?`
2. Если пользователь не найден — `abort(404)`.
3. Если `role == "admin"` — как и раньше: сообщение «Нельзя удалить мастер‑аккаунт», редирект.
4. **Если `archived_at` равен `NULL`** — удаление запрещено: закрыть соединение, `flash("Удалять можно только заархивированного пользователя. Сначала архивируйте.")` и `redirect(url_for("admin_users"))`.
5. Только если пользователь архивирован и не admin — выполнять `DELETE FROM users WHERE id = ?` (и при необходимости явную очистку `class_students` и `submissions`, либо полагаться на каскадное удаление, если оно задано в схеме).

Пример блока проверок:

```python
u = conn.execute(
    "SELECT id, username, role, archived_at FROM users WHERE id = ?",
    (user_id,),
).fetchone()
if u is None:
    conn.close()
    abort(404)
if u["role"] == "admin":
    conn.close()
    flash("Нельзя удалить мастер‑аккаунт.")
    return redirect(url_for("admin_users"))
if u["archived_at"] is None:
    conn.close()
    flash("Удалять можно только заархивированного пользователя. Сначала архивируйте.")
    return redirect(url_for("admin_users"))

# далее: DELETE FROM users WHERE id = ? и commit (очистка class_students/submissions — по схеме или явно)
```

### Шаг K3.2. Очистка связанных данных

Если в `db.py` для таблиц `class_students` и `submissions` заданы внешние ключи на `users(id)` с `ON DELETE CASCADE`, то при `DELETE FROM users WHERE id = ?` связанные строки удалятся автоматически. Тогда дополнительная очистка не нужна. Если каскада нет — перед удалением пользователя выполните явно:

```python
conn.execute("DELETE FROM class_students WHERE student_id = ?", (user_id,))
conn.execute("DELETE FROM submissions WHERE student_id = ?", (user_id,))
conn.execute("DELETE FROM users WHERE id = ?", (user_id,))
conn.commit()
```

### Шаг K3.3. Кнопка «Удалить навсегда» только для архивированных

В Части D кнопка «Удалить навсегда» показывалась для любого пользователя с ролью не admin. Теперь её нужно показывать **только** для пользователей, у которых задан `archived_at` (и не admin). В `admin_users.html` замените условие вывода формы удаления на:

```html
{% if u.archived_at and u.role != 'admin' %}
  <form method="post" action="{{ url_for('admin_user_delete', user_id=u.id) }}" class="inline-form">
    <button type="submit" class="btn btn-sm btn-danger"
            onclick="return confirm('Удалить пользователя безвозвратно? Это действие нельзя отменить.')">
      Удалить навсегда
    </button>
  </form>
{% endif %}
```

В итоге у активного пользователя есть только «Архивировать», у архивированного — «Восстановить» и «Удалить навсегда»; у мастера нет ни архивации, ни удаления.

## K4. Ограничение роли менеджера при создании пользователя

Только мастер (роль `admin`) может создавать пользователей с ролью «менеджер». Менеджер (роль `manager`) не должен иметь возможности выбрать роль «менеджер» (и по соглашению проекта — и роль «мастер»). Это новое поведение, его не было в Части D.

### Шаг K4.1. Проверка на бэкенде в `admin_user_create_view()`

В `views_admin.py` в функции создания пользователя:

1. Получите роль из формы: `role = (request.form.get("role") or "student").strip().lower()`.
2. Импортируйте `current_user` из `auth_utils` (если ещё не импортирован) и получите текущего пользователя.
3. Если текущий пользователь — менеджер (`current_user()["role"] == "manager"`) и выбранная роль — `"manager"`, подмените роль на `"student"` (или отклоните создание с сообщением) и при желании выведите flash.

Пример после получения `role` и проверок логина/пароля, до вызова `create_user`:

```python
from auth_utils import is_logged_in, is_admin, get_registration_open, create_user, current_user

# внутри admin_user_create_view():
role = (request.form.get("role") or "student").strip().lower()
# ... проверки username, password ...
cur = current_user()
if cur and cur["role"] == "manager" and role == "manager":
    flash("Менеджер не может назначать роль «менеджер». Создан пользователь с ролью «ученик».")
    role = "student"
# далее: allowed_roles и create_user(username, password, role)
```

### Шаг K4.2. Скрытие опций «Менеджер» и «Мастер» в форме для менеджера

В шаблоне `admin_users.html` в форме «Создать пользователя» в выпадающем списке ролей опции «Менеджер (manager)» и «Мастер (admin)» должны быть видны только мастерам. В контексте шаблонов уже есть `is_master()` (через `app.context_processor`). Сделайте список ролей так, чтобы при входе под менеджером отображались только «Ученик» и «Учитель»:

```html
<select id="new_role" name="role">
    <option value="student">Ученик (student)</option>
    <option value="teacher">Учитель (teacher)</option>
    {% if is_master() %}
    <option value="manager">Менеджер (manager)</option>
    <option value="admin">Мастер (admin)</option>
    {% endif %}
</select>
```

Менеджер при создании пользователя видит только «Ученик» и «Учитель»; мастер видит все четыре роли.
